아키텍처 접근법 3가지

Tauri의 Rust 백엔드와 React 프론트엔드 경계를 어떻게 나눌 것인가

A

React-Heavy (Thin Tauri)

Markdown 파싱, TOC, 프로젝트 분석 전부 JS

장점

  • 개발 속도 빠름
  • JS 생태계 풀 활용
  • Tauri 지식 최소화

단점

  • 대용량 파일 처리 느림
  • 메모리 사용량 높음
  • 10k+ 문서에서 병목

Rust 담당: FS 접근, OS 다이얼로그만

B

Rust-Heavy (Thick Backend) ⚠

파싱/분석/DB 전부 Rust, React는 렌더만

장점

  • 최고 성능
  • 메모리 효율
  • 대규모 프로젝트 대응

단점

  • 개발 속도 느림
  • Rust 구현 복잡
  • Mermaid/Shiki는 JS만 지원

JS가 해야만 하는 것들(Mermaid, Shiki)로 인해 브리지 복잡도 증가

C

Hybrid — 권장 ✨

명확한 Rust/JS 경계, 각자 잘하는 것만

🦀 RUST
  • 파일 시스템 / 감시
  • SQLite (상태 저장)
  • 프로젝트 유형 분석
  • 문서 인덱싱
  • 파일 메타데이터
⚛️ REACT/JS
  • Markdown 렌더링 (remark)
  • Mermaid.js 다이어그램
  • Shiki 코드 하이라이트
  • TOC 파싱/탐색
  • UI 상태 (Zustand)

장점

  • 실용적 성능/생산성 균형
  • JS 전용 라이브러리 그대로 사용
  • FS 작업은 Rust가 안전하게

단점

  • Rust/JS 경계 설계 필요
  • Tauri Command API 학습

권장: C — Hybrid. Mermaid와 Shiki는 JS 생태계에 강결합되어 있어 Rust에서 처리 불가. 동시에 파일 시스템 작업과 대규모 인덱싱은 Rust가 훨씬 안정적. 경계가 명확하면 개발도 빠름.
